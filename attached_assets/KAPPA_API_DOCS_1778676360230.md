## Heartbeat

### `POST /heartbeat`

Send a heartbeat from a device. Updates the device's online status and last heartbeat timestamp.

**Request Body:**
```json
{
  "deviceId": "echo-pc",       // required
  "metadata": {                 // optional, arbitrary system metrics
    "cpu": 45.2,
    "memory": 62.1,
    "disk": 78.5,
    "uptime": 86400,
    "clientTimestamp": 1775060400000  // optional — if provided, server computes latency
  }
}
```

**Latency calculation:** If `metadata.clientTimestamp` is a Unix ms timestamp, the server computes `latencyMs = serverTime - clientTimestamp` and stores it on the heartbeat record. The server also computes `_intervalMs` (time since last heartbeat) and enriches the stored metadata with both values.

**Response (200):**
```json
{
  "ack": true,
  "serverTime": 1775060400000,     // Unix timestamp ms
  "nextExpectedMs": 15000          // Expected interval until next heartbeat
}
```

### `GET /heartbeat/:deviceId/jitter`

Analyze heartbeat interval stability and latency statistics for a device. Useful for detecting unstable connections.

**Query Parameters:**
| Param   | Type    | Default | Description              |
|---------|---------|---------|--------------------------|
| `hours` | integer | 1       | How many hours to analyze |

**Response (200):**
```json
{
  "deviceId": "echo-pc",
  "sampleCount": 51,
  "intervalStats": {
    "expectedMs": 15000,
    "meanMs": 15200,
    "stdDevMs": 1250,
    "minMs": 12000,
    "maxMs": 22000,
    "coefficientOfVariation": 8.2
  },
  "latencyStats": {
    "meanMs": 45.2,
    "stdDevMs": 12.1,
    "minMs": 22.0,
    "maxMs": 98.5,
    "sampleCount": 48
  },
  "stable": true
}
```

**Notes:**
- `stable` is `true` if the coefficient of variation is below 50% (configurable via `JITTER_ALERT_THRESHOLD` env var)
- `intervalStats` shows how consistent the heartbeat intervals are (lower stdDev = more stable)
- `latencyStats` is only populated when agents send `clientTimestamp` in heartbeat metadata
- `coefficientOfVariation` is `(stdDev / mean) * 100` — values above ~30% indicate connection instability

### `POST /thresholds`

Create a new sensor threshold.

**Request Body:**
```json
{
  "deviceId": null,              // null = applies to ALL devices; or specific device ID
  "sensorType": "temperature",   // required — matches sensor_readings.sensor_type
  "metricKey": "celsius",        // required — the key inside the sensor values JSON
  "minValue": 5,                 // optional — alert if value drops below this
  "maxValue": 80,                // optional — alert if value exceeds this
  "severity": 3                  // optional, default 2 (1=info, 2=warning, 3=critical)
}
```

**Response (200):** The created threshold object.

**Notes:**
- At least one of `minValue` or `maxValue` is required
- The background job checks thresholds every 30 seconds against the latest sensor readings
- Duplicate alerts are suppressed for 5 minutes per sensor/metric combination

---

### `PUT /thresholds/:id`

Update an existing threshold's values or severity.

**Request Body:**
```json

  "minValue": 10,
  "maxValue": 90,
  "severity": 2
}
```

**Response (200):** The updated threshold object.

Downloadable agent scripts that devices run to connect to the KAPPA server.

Returns a Python agent script for desktop (Windows/Linux/Mac). Uses `psutil` and `websocket-client`. Content-Type: `text/plain`.

Returns a Python agent script for Termux (Android). Uses `termux-api` commands for battery, wifi, location data. Content-Type: `text/plain`.

Returns a minimal bash script using `curl` to register and send heartbeats via the REST API. Content-Type: `text/plain`.

## WebSocket Protocol

**Endpoint:** `wss://<your-domain>/ws?deviceId=<device-id>`

The WebSocket connection provides real-time bidirectional communication between agents and the server.

### Client → Server Messages

#### Register
```json
{
  "type": "register",
  "name": "My Device",
  "deviceType": "pc",
  "os": "windows",
  "capabilities": ["cpu", "memory", "disk"],
  "metadata": { "hostname": "ECHO-PC" }
}
```
**Server reply:**
```json
{
  "type": "registered",
  "deviceId": "echo-pc",
  "serverTime": 1775060400000,
  "heartbeatIntervalMs": 15000
}
```

#### Heartbeat
```json
{
  "type": "heartbeat",
  "cpu": 45.2,
  "memory": 62.1,
  "disk": 78.5
}
```
**Server reply:**
```json
{
  "type": "heartbeat_ack",
  "serverTime": 1775060400000
}
```

#### Sensor Data
```json
{
  "type": "sensors",
  "readings": [
    {
      "sensorType": "temperature",
      "timestamp": 1775060400000,
      "values": { "celsius": 42.5 }
    }
  ]
}
```

#### Alert
```json
{
  "type": "alert",
  "message": "High CPU usage detected",
  "severity": 2,
  "metadata": { "cpu_percent": 98.5 }
}
```

### Server → Client Messages

#### Command (sent via `POST /kappa/command`)
```json
{
  "type": "command",
  "command": "collect_sensors",
  "args": {}
}
```

### Connection Lifecycle
1. Client connects to `wss://<domain>/ws?deviceId=my-device`
2. Client sends a `register` message
3. Server replies with `registered` including the `heartbeatIntervalMs`
4. Client sends `heartbeat` messages at the specified interval
5. On disconnect, server marks device as offline and records `lastDisconnect`
6. If a device was previously offline, a `reconnect` alert (severity 1) is created on re-registration

## Background Jobs

### Heartbeat Monitor (every 15 seconds)
- Checks all devices marked `online`
- If `last_heartbeat` is older than 45 seconds (configurable via `HEARTBEAT_TIMEOUT_MS` env var), marks device as offline
- Creates a severity 2 (warning) `timeout` alert for the device
- For online devices, checks heartbeat **jitter** (interval standard deviation) and creates a severity 2 `jitter` alert if coefficient of variation exceeds threshold (default 50%, configurable via `JITTER_ALERT_THRESHOLD`)
- Jitter alerts are suppressed for 10 minutes to avoid spam

### Sensor Threshold Monitor (every 30 seconds)
- Evaluates all configured sensor thresholds against the latest sensor readings (last 2 minutes)
- Supports both min and max bounds per sensor type and metric key
- Thresholds can be global (all devices) or device-specific
- Creates alerts with configurable severity when values breach thresholds
- Duplicate threshold alerts suppressed for 5 minutes per sensor/metric combination
- Alert type: `threshold`, with metadata including the threshold ID, actual value, and bounds

### Uptime Rollup (every 5 minutes)
- Aggregates heartbeat data into hourly buckets in the `uptime_history` table
- Computes per-bucket: heartbeat count, estimated online seconds, uptime %, average/max latency, average jitter
- Upserts (updates if bucket already exists for the current hour)
- Retained for 30 days (cleaned up by the cleanup job)

### Data Cleanup (every 1 hour)
- Deletes heartbeat records older than 24 hours
- Deletes sensor readings older than 6 hours
- Deletes acknowledged alerts older than 7 days
- Deletes uptime history older than 30 days

## Quick Start (Connecting a Device)

### Via REST API (simplest)
```bash
# Register
curl -X POST https://<domain>/api/devices/register \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"my-pc","name":"My PC","type":"pc","os":"linux"}'

# Send heartbeat (include clientTimestamp for latency tracking)
curl -X POST https://<domain>/api/heartbeat \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"my-pc","metadata":{"cpu":25.0,"memory":40.0,"clientTimestamp":'$(date +%s%3N)'}}'
```

### Via WebSocket (recommended for persistent connections)
```javascript
const ws = new WebSocket("wss://<domain>/ws?deviceId=my-pc");

ws.onopen = () => {
  ws.send(JSON.stringify({
    type: "register",
    name: "My PC",
    deviceType: "pc",
    os: "linux",
    capabilities: ["cpu", "memory"]
  }));

  setInterval(() => {
    ws.send(JSON.stringify({
      type: "heartbeat",
      cpu: Math.random() * 100,
      memory: Math.random() * 100
    }));
  }, 15000);
};
```
