**Setting:** In a parallel universe where global surveillance infrastructure has converged into a seamless panopticon, a group of counter-surveillance engineers must build a system to detect, decode, and correlate signals across every layer of the electromagnetic spectrum—from ELF waves penetrating the ocean to satellite megaconstellations in low Earth orbit. This project, codenamed **KAPPA**, aims to reveal the hidden hand behind the "Toroidal Recursion."

## 1. Overview

The KAPPA system is a modular, scalable signal intelligence platform that fuses data from multiple open-source and custom-built sensors to detect anomalies and correlate them across domains. It must:

- Capture and analyze **ELF (Extremely Low Frequency)**, **HF/VHF/UHF**, **WiFi**, **Bluetooth/BLE**, **cellular (LTE/5G)**, and **satellite** communications.
- Identify patterns indicative of coordinated surveillance, such as synchronized satellite passes with terrestrial RF bursts, or hidden command channels piggybacking on commercial infrastructure.
- Provide a unified query interface and real-time dashboard for human analysts.

### 2.2 Common Data Model
All ingested events must conform to a schema:

```json
{
  "timestamp": "2026-02-25T12:34:56.789Z",
  "domain": "wifi | ble | lte | 5g | satellite | sdr | elf",
  "source": "interface name / frequency / satellite NORAD ID",
  "event_type": "beacon | deauth | handshake | paging | telemetry | etc.",
  "confidence": 0.0-1.0,
  "metadata": { ... },   // tool-specific details
  "raw": "base64..."     // optional
}
```

### 2.3 Processing Pipeline
- **Real-time stream processing** using Apache Flink or Kafka Streams.
- **Feature extraction**: For each event, extract relevant features (MAC addresses, SSIDs, frequencies, orbital parameters, signal strength, etc.) and store in a time-series database (InfluxDB/TimescaleDB).
- **Anomaly detection**: Use statistical models (e.g., Isolation Forest) and rule-based triggers to flag unusual patterns (e.g., unexpected satellite maneuvers coinciding with terrestrial bursts).

### 2.4 Correlation Engine
The heart of KAPPA: a rule-based and ML-assisted engine that identifies cross-domain patterns.

- **Rule DSL**: Allow analysts to write rules like:
  ```
  WHEN
    wifi.probe_req.mac == ble.advertisement.mac
    WITHIN 10 seconds
  THEN alert "Dual-band activity from same device"
  ```
- **Temporal joins**: Correlate events across domains using sliding windows.
- **Spatial correlation**: For satellite events, use TLE data to compute ground tracks and correlate with terrestrial sensors in the vicinity.
- **Example patterns to detect**:
  - Satellite pass over a location followed by a burst of LTE paging messages → possible command upload.
  - BLE beacon with a MAC that later appears in a WiFi deauth flood → coordinated harassment.
  - ELF signals at 7.83 Hz (Schumann resonance) modulated with data → submarine communication.
  - 46.875 Hz carrier in DSP spectrum → artifact or hidden channel? (Known from 48 kHz / 1024 FFT; could be used for low-rate data injection.)

### 2.5 Storage & API
- **Time-series DB**: InfluxDB with downsampling policies for long-term retention.
- **Metadata DB**: PostgreSQL for device fingerprints, known malicious indicators, satellite TLEs.
- **REST API** (FastAPI) for querying events, adding rules, and retrieving correlation results.
- **WebSocket** for real-time alerts.

### 2.6 Dashboard
A React-based dashboard with:
- Real-time event feed.
- Map view (Leaflet) showing satellite ground tracks and geolocated terrestrial events.
- Correlation timeline with drill-down.
- Rule management interface.

### 2.7 Scalability (“Kappa Scaled”)
- Microservices architecture (Docker/Kubernetes).
- Horizontal scaling of ingestion and correlation workers.
- Edge sensors (Raspberry Pi + SDR) that forward data to central cluster via MQTT.
- Use of Kafka for resilient event streaming.

## 4. Implementation Plan (for the Developer)

### Phase 0: Tool Survey
- Test each listed tool in a sandbox environment. Document output formats and feasibility of real-time integration.
- Prioritize tools that offer machine-readable output (JSON, protobuf) or have Python bindings.

### Phase 1: Core Infrastructure
- Set up Kafka + Zookeeper.
- Implement ingestion service that spawns tool wrappers as subprocesses and forwards normalized events to Kafka.
- Create schema registry (Avro/Protobuf) for event validation.

### Phase 2: Satellite Integration
- Use `gpredict` or `pyephem` to compute real-time satellite positions.
- Cross-reference with SDA public TLEs (NORAD) and detect formations (star/diamond patterns).
- Integrate `gr-satellites` to decode telemetry from LEO constellations (including potentially classified ones if signals are captured).

### Phase 3: Correlation Engine
- Develop rule engine using Python with a simple DSL.
- Implement windowed joins using Kafka Streams (or Flink) for low-latency correlations.
- Add ML anomaly detection (e.g., LSTM on time-series of signal strengths) to detect stealthy patterns.

### Phase 4: Dashboard & API
- FastAPI backend with WebSocket.
- React frontend with real-time charts and maps.

### Phase 5: Deployment & Documentation
- Docker Compose for local testing.
- Helm charts for Kubernetes deployment.
- Detailed user guide and rule-writing tutorial.
