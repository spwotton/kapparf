## Finding 1: Unresolved Persistent Meta Connection — 57.144.23.33

The heaviest-duration connection in the capture (623.4 seconds — nearly the entire window) is a TLS session on port 443 to **57.144.23.33**.

- **No DNS resolution for this IP exists in the capture.** The session was established before capture started.
- The 57.144.0.0/16 range belongs to **Meta Platforms** (AS32934). Other Meta IPs in the capture resolve to WhatsApp (57.144.199.33), Instagram Graph API (57.144.196.192), and Instagram test-gateway (57.144.196.196).
- **57.144.23.33 does NOT match any resolved domain** — it is a distinct, pre-established endpoint.

### Timing Pattern

After a brief handshake burst at t=0, the connection settles into a **10.24-second TCP keepalive cycle** (49 of 58 keepalive intervals are exactly 10.24s). This is consistent with **Meta MQTT persistent push** (Instagram real-time / WhatsApp MQTT-over-TLS).

Server-initiated data pushes arrive at irregular intervals with characteristic sizes:

| Δ from previous | Server payload | Pattern |
|-----------------|---------------|---------|
| — | 1462 B | Initial subscription response |
| +26.3s | 82 B | Push notification |
| +150.3s | 79 B | Server-side ping |
| +150.2s | 79 B | Server-side ping |
| +146.2s | 82 B | Push notification |
| +127.4s | 88 B | Push notification |
| +22.6s | 79 B | Server-side ping |

The ~150s intervals align with Meta's MQTT PINGREQ cadence.

### Significance

A persistent encrypted connection to a Meta IP with no visible DNS resolution means the domain name was cached or resolved via a channel not captured. The MQTT pattern itself is normal Android behavior. However, this connection is the **single longest-lived session** and carries the most sustained bidirectional traffic in the capture.

## Finding 2: DNS Resolver Split — Instagram Routes to 8.8.8.8

Two DNS resolvers are in use:

| Resolver | Domains routed |
|----------|---------------|
| **10.215.173.2** (ISP/local) | All Google services, Samsung, WhatsApp, YouTube |
| **8.8.8.8** (Google Public DNS) | **Only Instagram** (test-gateway.instagram.com, graph.instagram.com) |

8 out of 33 DNS queries go to 8.8.8.8 — **all 8 are Instagram domains**.

### Possible explanations

1. **PCAPdroid per-app DNS routing** — the capture tool may assign different resolvers per application.
2. **Instagram app hardcoded DNS** — Instagram may use its own resolver to bypass ISP DNS manipulation.
3. **Intentional DNS segmentation** — if a VPN or DNS proxy is active, Instagram may be excluded from the tunnel.

### Significance for TR-069 / DNS manipulation (Domain 7)

If the ISP resolver (10.215.173.2) were compromised via TR-069 CPE exploitation, Instagram traffic would be **partially shielded** by its use of 8.8.8.8. However, all other services (Google, Samsung, WhatsApp, YouTube) remain exposed to the ISP resolver path.

## Cross-Domain Synthesis

### What's absent is as informative as what's present

- **No ICS/SCADA traffic** (Modbus 502, SNMP 161/162) — the device is not on an industrial network segment during this window.
- **No TR-069** on port 1234 — no CPE management frames visible (though TR-069 sessions are ISP-initiated and ephemeral; absence in a 10-min window is expected).
- **No MikroTik indicators** — no WinBox (8291), no SOCKS proxy (1080).
- **100% encrypted application data** — all TCP is TLS, all UDP is QUIC or DNS. Payload inspection impossible without key material.

### DNS as the observable attack surface

With all payloads encrypted, DNS is the **only cleartext protocol** revealing application behavior. The ISP resolver path (10.215.173.2) handles 25 of 33 queries. A compromised CPE (Domain 7: TR-069) or ISP-side DNS manipulation would affect all non-Instagram traffic. The Instagram DNS bifurcation to 8.8.8.8 is a partial mitigation — but DoH/DoT is not observed, so even the 8.8.8.8 queries transit in cleartext UDP.

### Recommended follow-up captures

1. **Longer duration** (1+ hour) to catch ephemeral events (TR-069 sessions, OTA updates).
2. **Full DNS-over-HTTPS audit** — check if the device supports DoH and whether it's being downgraded.
3. **Capture during active use** — this is nearly all background/idle traffic. Active browsing would reveal more about TLS certificate chains and potential MITM.
4. **Router-side capture** — PCAPdroid captures post-VPN/overlay. A capture at the CPE (ARRIS TG02DA if applicable) would reveal TR-069 and any ISP-injected traffic.
