LACK PAPER — PROJECT BREAKWATER
Architectural Specification v0.1 | Node #1090 | May 2026
ABSTRACT

A new class of publication infrastructure that is simultaneously a credentialed news outlet, an interactive evidence graph engine, a client-side AI inference mesh, and a zero-knowledge whistleblower ingestion system. The architecture proves — by demonstration rather than claim — that open-source, in-browser computation can replace server-dependent surveillance-vulnerable platforms for sensitive journalism. The GOS mathematical substrate provides the graph topology. The hypervisor AI layer provides context routing. The news site provides legal and editorial cover that is also real.
PART I: THE PROBLEM WITH EXISTING PLATFORMS

Every current whistleblower platform fails on one of three axes:

    Server trust: SecureDrop, Signal, ProtonMail all require trusting a server that can be subpoenaed, seized, or silently compromised. The server knows who connected.
    Linear narrative: Journalism CMS (WordPress, Ghost, Substack) renders stories as sequential text. Evidence is non-linear. A document, a frequency, a corporate registry entry, and a bank transaction are nodes in a graph — forcing them into paragraphs destroys the topology.
    Passive consumption: Readers cannot contribute connections. The journalist is the only graph editor. Verified crowdsourced graph-building is architecturally impossible in every existing system.

Breakwater solves all three by moving the trust boundary into the browser.
PART II: ARCHITECTURE LAYERS
Layer 0 — The Surface (News Site)

A real editorial publication with ISSN registration, editorial standards, and published content. Not a cover — a genuine outlet. The credentialing matters because:

    Press freedom law applies
    Shield law protections apply in relevant jurisdictions
    Source relationships are legally protected
    The site can be indexed, cited, and referenced in courts

Stack: Static-first React/Vite build. Content in MDX. Design system from attached tokens (EB Garamond for editorial, Montserrat for UI, emerald #059669 accent, dark mode via .dark CSS class). Deployed to a CDN with no origin server storing user behavior.
Layer 1 — The Evidence Graph Engine (In-Browser)

Every article has an associated graph of evidence nodes. Nodes are typed:

NodeType: DOCUMENT | FREQUENCY | CORPORATE | FINANCIAL | PERSON |
          PROPERTY | TIMESTAMP | LOCATION | COMMUNICATION | RF_CAPTURE

Edges are typed:

EdgeType: OWNS | CONTROLS | FUNDED_BY | CO-LOCATED | TEMPORAL_MATCH |
          HARMONIC_OF | CONTRACTS_WITH | LEGALLY_CONNECTED | UNKNOWN

The graph renders in-browser using a WebAssembly force-directed layout engine (no server). Users can:

    Pan and zoom the evidence graph
    Click nodes to expand source documents
    Draw proposed edges (submitted anonymously, held for editorial review)
    See confidence scores on each edge derived from source count × cross-domain verification

The GOS Antikythera mod-24 spoke table provides the base topology — 24 primary sectors, each sector containing nodes from a specific domain layer. This is not decorative. It's the actual graph partitioning algorithm, and it matches the 13-dimensional toroidal manifold topology that makes the clustering mathematically coherent.
Layer 2 — Client-Side AI Inference (The Hypervisor)

No API calls to external AI services. All inference runs in-browser via:

    transformers.js (Xenova) for embedding and semantic similarity
    WebLLM (MLC) for local LLM inference on capable hardware
    Fallback: smaller quantized models (GGUF Q4) loaded from CDN via range requests

What the AI layer does:

    Context routing: when a user views a node, the AI computes semantic distance to all other nodes and surfaces the 5 most relevant non-obvious connections
    Gap detection: identifies nodes with high connectivity potential but low current edge count — these are the open questions, surfaced explicitly
    Narrative synthesis: on request, generates a plain-language summary of a subgraph, citing only the nodes within it
    Anomaly flagging: statistical analysis of temporal patterns (the 46.875 Hz / frequency correlation engine, generalized) — any evidence set can be run through the f_Atlas correlation detector

All inference is local. The server never sees what the user is analyzing. The server only stores the published editorial graph — the user's exploration is private.
Layer 3 — Anonymous Ingestion (Zero-Knowledge Contribution)

Whistleblower submission via a system that never creates a server-side association between submitter and submission:

    Submitter opens the in-browser ingestion tool (no login, no cookies)
    Document is encrypted in-browser with the publication's public key (Ed25519)
    A zero-knowledge proof is generated: "this submitter has a valid document meeting the schema, without revealing the document or the submitter"
    Only the encrypted blob and the ZK proof reach the server
    Editorial team decrypts offline with an air-gapped key

The ZK proof system uses snarkjs (Groth16 proofs) compiled to WASM — runs entirely in browser. Proof generation takes 2–8 seconds on a standard device.

Onion routing for the upload: the ingestion endpoint is also reachable as a Tor hidden service (.onion). The CDN and the .onion are the same editorial backend — users choose their threat model.
Layer 4 — The Open-Source Proof Layer

The entire system is public, auditable, and forkable. This is the demonstration:

    Every graph algorithm is open source and verifiable
    The GOS constants (κ₁, κ₂, f_Atlas, φ, Δ) are the documented mathematical basis for the graph topology — published in the ANVIL, citable
    The AI inference models are publicly available quantized open-weights models
    The ZK circuits are published and reproducible
    Any journalist or researcher can fork the entire system and run their own instance

The message: surveillance-resistant journalism infrastructure is now a client-side mathematics problem, not a server trust problem. The math is the protection.
PART III: THE WHISTLEBLOWER PROTECTION ARCHITECTURE IN PLAIN LANGUAGE

The platform is protective not because it promises privacy — it's protective because the architecture makes it structurally impossible to log what users think:
What the server knows	What the server does NOT know
Which articles were loaded	Which nodes the user examined
That a ZK proof was submitted	Who submitted it or what it contained
That the graph was rendered	What connections the user drew
Static asset delivery	Any inference result or AI interaction

The gap between those two columns is where source protection lives. It's not a policy. It's a geometry.

This is both the test case and the proof of concept. It demonstrates the graph engine on real data. It demonstrates the AI context routing on a real non-linear evidence set. And it is a real published story.
PART V: TECHNICAL BUILD SPECIFICATION

Frontend: React + Vite, pnpm workspace, TypeScript strict. Design tokens as specified in attached document. Dark mode first.

Graph engine: Cytoscape.js (WASM-accelerated layout) or D3-force for the physics layer. Custom React wrapper exposing the GOS node/edge type system.

AI layer: transformers.js for embeddings (all-MiniLM-L6-v2 as default), WebLLM for generation (Phi-3-mini-4k or Mistral-7B-Q4 depending on device capability). Graceful degradation to semantic search only on low-memory devices.

ZK layer: snarkjs Groth16. Circuits written in Circom. Trusted setup ceremony using a public multi-party computation (Phase 2 Powers of Tau).

Ingestion backend: Minimal Express server accepting only encrypted blobs + ZK proofs. No logging middleware. No IP storage. Returns only a receipt hash.

Deployment: Static frontend to Cloudflare Pages. Ingestion backend to a jurisdiction with strong press shield laws (Iceland, Switzerland, or Netherlands). Tor hidden service mirroring the ingestion endpoint.

This is a journalism infrastructure project. It is also a mathematics demonstration. It is also a statement that the GOS framework's toroidal graph topology is a better model for how evidence actually connects than anything currently deployed in investigative journalism tooling.

The architecture is the argument.
